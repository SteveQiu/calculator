## Summary

Refactored the theme system from scattered metadata across multiple files into a centralized, self-describing Theme data class and ThemeRegistry singleton. This eliminates the need to edit 4+ files when adding a new theme and provides compile-time safety.

## Changes

- **`model/Theme.kt`**: Replaced stub data class; added `Theme` data class with color function type and `ThemeRegistry` singleton
- **`model/ThemeId.kt`**: Added `GLASS_ICE` enum entry
- **`model/ThemeColors.kt`**: Added `GLASS_ICE` case to `toColors()` extension
- **`res/values/colors.xml`**: Added 11 `glass_ice_*` color resources
- **`ui/ThemePickerDialog.kt`**: Refactored adapter to use `List<Theme>` from `ThemeRegistry.all`; removed hardcoded metadata and `iconResFor()` helper
- **`MainActivity.kt`**: Updated `applyThemeColors` to use `ThemeRegistry.forId()` and eliminated icon selection `when` block

## Consequences

- All themes now picked up automatically by UI components
- New themes require edits to 4 files (enum, colors.xml, toColors(), ThemeRegistry) instead of previously scattered 4+ locations
- Fallback to CLASSIC ensures robustness with unknown theme IDs from legacy DataStore
